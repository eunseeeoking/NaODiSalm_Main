## 2. ODsay LAB 일일 호출 게이트 (800회/1,000회)

### 배경
- ODsay 무료 한도 1,000건/일. 추천 카드 1건 당 ODsay 1회 호출 → 사용자 ~20명만 들어와도 폭주 가능.
- 한도 초과 시 ODsay 측에서 일괄 차단 → 추천 화면 전부 깨질 위험.

### 구현
| 파일 | 역할 |
|------|------|
| `server/prisma/schema.prisma` (model `OdsayUsageDaily` / `t_odsay_usage_daily`) | KST 일자별 호출량 row (`date` PK, `callCount`) |
| `server/src/services/external/odsayQuota.ts` (신규) | 게이트 모듈 — `checkAndConsumeOdsayQuota()`, `refundOdsayQuota()`, `getOdsayUsageToday()`, `ODSAY_DAILY_LIMIT = 800` |
| `server/src/services/external/odsay.ts` | `fetchOdsayRoute` 진입부에 게이트 추가. 차단 시 `null` 반환 → 호출자(`commuteRepository`) 는 Haversine 폴백 |
| `server/src/routes/domains/admin.ts` | `GET /api/admin/odsay-usage` — 오늘 호출량/잔량/차단여부 조회 |
| `server/prisma/migrations/2026050715xxxx_add_odsay_usage_daily/` | (예정 — 현재 마이그레이션 실패, 아래 §6 참조) |

### 정책 핵심
- **차단 임계값 800** = 무료 한도 1,000 × 0.8 (20% 마진).
- **카운트 단위**: 실제 ODsay 외부 호출 1건당 +1. 캐시 hit, 게이트 차단, `-98/-99` 비즈 에러는 미카운트.
- **환불**: `fetch` 자체 reject(네트워크 오류) 시에만 `-1`. HTTP 200 + 에러 응답은 환불하지 않음 (ODsay 측에서 이미 카운트했을 가능성).
- **리셋**: KST 자정 — `todayKstYmd()` 가 UTC+9 시프트로 처리. cron 불필요(키 변경 시 새 row).
- **동시성**: 단일 서버 가정. `read → check → increment` 사이 미세 race 허용 (±5건은 800 마진 안에서 흡수). 격하면 `SELECT … FOR UPDATE` 로 보강 가능.

### Steelman
- "초과 시 환불 후 false 반환은 0건 추가 부담인데 굳이?" → upsert + increment 가 단일 트랜잭션이라 race 안전. 환불 패턴이 가장 단순함.
- "왜 1000 이 아니라 800?" → 가짜 카운트(중복 호출/잘못된 환불 누락)로 인한 over-count 마진. ODsay 가 키 차단을 걸면 복구가 24h+ 걸리므로 보수적으로.

## 4. LH 매물 미표시 — 원인 진단 (수정 없음, 후속 작업으로 분리)

### 증상
`/region-detail?legalDongCode=…` 진입 시 "LH 국가" 탭 카운트가 **항상 0**. mock 데이터(당산·충정로·망원·신도림) 환경에선 보였으나 실 DB 환경에선 누락.

### 4-Way 분석

| # | 가설 | 검증 | 결론 |
|---|------|------|------|
| A | `isLhComplex` 필드가 응답에 안 들어옴 | `regions.ts:266` 에서 `isLhComplex: isLhByName(c.name)` 항상 세팅 | 필드 자체는 OK |
| B | `isLhByName` 키워드 매칭 실패 | `t_apt_complex` 는 **MOLIT 실거래가** 출처 — LH 임대주택은 매매 거래가 없어서 이 테이블에 거의 없음. 단지명이 "행복주택"/"LH" 인 row 0건 추정 | **루트 원인 #1** |
| C | LH 데이터는 `t_lh_youth_housing` 에 따로 적재되는데 `regions.ts` 가 이걸 join 안 함 | grep: `regions.ts` 는 `aptComplex` 만 select. `lhYouthHousing` 은 `recommendationRepository.ts:354` 의 카운트 집계에만 사용 | **루트 원인 #2** |

### 처방 (제안 — 별도 PR 권장)

1. **즉시 (낮은 위험)**: `regions.ts` 에 `t_lh_youth_housing` join 추가.
   - sigunguCode(5자리) 기준 LH 단지를 별도 카드로 합쳐서 반환.

2. **중기 (스키마 변경)**: `t_lh_youth_housing.legal_dong_code` 를 시군구 5자리 → 행정동 10자리로 정규화.

3. **운영**: `.env` 에 `MOLIT_SERVICE_KEY` 가 실제로 들어가 있는지 확인. 미설정 시 `lhClient` 가 warn 로그 1회 + 빈 배열. 현재 환경에서 이 로그 떨어지는지 first check.

## 5. 수도권 거래내역 추가 — 분석 (작업 미실행, 결정 보류)

### 현황
- DB 용량 **2.8 GB** (서울 25개 구 실거래 ~5년분 + LSTM training_result + 보조 테이블).
- 통근 폴리곤은 §1 처럼 수도권 전체로 확장됨 → 사용자가 경기/인천 행정동 클릭 가능.
- 그런데 추천 파이프라인(`recommendationRepository.ts:200` ~ `fetchRegionPrices`) 은 `t_apt_trade` JOIN `t_apt_complex` 만 보므로, 경기·인천 행정동은 **거래 0건 → 추천 카드 빈 결과**.

### 결정 트리

| 옵션 | 디스크 영향 | UX 효과 | 작업량 |
|------|-------------|---------|--------|
| A. 수도권 전체 실거래 풀 적재 | +6~8 GB (총 ~10 GB) | 가장 정확. 모든 행정동에서 완전 추천 | bulkIngest 5~10시간, MOLIT API 호출 한도 주의 |
| B. **시군구 단위 월별 통계만 적재** (avg_price, avg_area, count) | +200~400 MB | 추천 알고리즘에는 충분, 단지 카드(`regions.ts`) 는 빈 결과 그대로 | 신규 집계 테이블 1개 + 시드 스크립트 |
| C. lazy ingest — 사용자가 클릭한 행정동만 cron 으로 적재 | +1~2 GB (수요 지역만) | 첫 클릭 시 1~2분 빈 결과, 이후 정상 | 큐 인프라 필요, 복잡도 ↑ |
| D. 폴리곤 표시 영역을 서울로 되돌리고 거래내역도 서울만 유지 | 0 | 이미 확장한 폴리곤 의미 무화 | 0 |

### 권고
**B → A 순으로 단계 진행** 권장.
- 지금 단계(MVP/공모전 데모)에선 **B** 가 ROI 최선. 추천 점수만 채워지면 데모상 빈 화면 안 보임.
- 데모 후 사용자 트래픽이 일정 수준 넘으면 **A** 로 풀 적재.
- 디스크 2.8 → 10 GB 는 RDS/Cloud SQL 기준 월 비용 ~$50 추가. 백업 시간도 비례 증가.

### Steelman (B 안의 반대)
- "통계만 있으면 단지 카드(`/region-detail`) 는 여전히 비어 있어서 '클릭은 되는데 내용 없음' UX 깨짐" → 동의. **B 채택 시 클라이언트에서 거래내역 미적재 행정동은 폴리곤 회색 + 클릭 비활성** 같은 시각 피드백을 같이 깔아야 함.

## 6. 알려진 환경 이슈 — Prisma migrate dev 권한 실패

```
Error: P3006 / P3018 — Database error code: 1142
ALTER command denied to user 'molit'@'172.18.0.1' for table 't_user'
Migration: 20260507155826_add_password
```

- 원인: `migrate dev` 가 shadow DB 에 모든 마이그레이션 재적용 시도 → `molit` 계정에 `ALTER/CREATE/DROP` 권한 없음.
- §2 의 `add_odsay_usage_daily` 마이그레이션도 같은 이유로 적용 실패.
- 해결: (a) `GRANT ALL PRIVILEGES ON molit_contest.* TO 'molit'@'%';` + 글로벌 `CREATE` 권한, 또는 (b) `shadowDatabaseUrl` 환경변수 분리 후 shadow 전용 권한 계정. 자세한 처방은 별도 메모 참조.
- 현재 schema.prisma 의 `OdsayUsageDaily` 모델은 정의돼 있고, 운영 DB 에는 임시로 `prisma db push` 또는 수동 DDL 로 테이블만 생성하면 동작.

## 8. Phase 1.5 (revert) — LH 단지 카드 제거 + 시군구 집계 배너로 재설계

§7 Phase 1 에서 LH 를 단지 카드로 끼워넣은 게 잘못이라는 사용자 피드백. LH 데이터는 단지 디테일(이름·좌표·세대수)이 없어 가짜 "LH-{id}" 카드를 만든 셈. 시군구 집계로만 정직하게 노출하도록 재설계.

### Root Cause
- `t_lh_youth_housing` 의 데이터 입자가 시군구 5자리 + programType 집계 (lhLeaseInfo1 API 응답 한계). 단지 1건 = (시군구, 프로그램타입) 하나의 row.
- 단지명·좌표·세대수 없음 → 단지 카드 형태로 표현하면 모든 필드가 0 인 깡통 카드가 됨.
- 추가 발견: `recommendationRepository.ts:354` 의 `lhComplexNearby` count 도 10자리 후보 vs 5자리 저장값 매칭 시도로 **항상 0** 반환 중. Depth 2 의 "LH N" 배지도 작동 안 했음.
