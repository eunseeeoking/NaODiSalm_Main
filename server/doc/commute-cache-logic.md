# 통근 매트릭스 캐시 동작 로직

> 위치: `server/src/services/external/odsay.ts` → `makeCacheKey()`
> 테이블: `t_commute_matrix` (cacheKey + legalDongCode 복합 unique)

## 핵심 요약 한 줄

> 직장의 (위도, 경도) 를 **소수점 4자리까지 반올림** 해서 캐시 키로 쓴다.
> 같은 키 = 같은 캐시 = ODsay 호출 0.

## 캐시 키 생성

```ts
function makeCacheKey(lat: number, lng: number): string {
  return `${lat.toFixed(4)}_${lng.toFixed(4)}`;
}
```

예시:

| 입력 | cacheKey |
|---|---|
| (37.49792, 127.02764) | `37.4979_127.0276` |
| (37.49788, 127.02755) | `37.4979_127.0276`  ← 같은 키 (4자리 동일) |
| (37.49850, 127.02780) | `37.4985_127.0278`  ← 다른 키 |

## 격자 크기 — 4자리 반올림이 의미하는 것

소수점 4자리 = `0.0001` 단위로 그룹화.

| 축 | 1도 거리 | 0.0001 도 거리 (서울 위도 37°) |
|---|---|---|
| 위도 | 약 111 km | **약 11.1 m** |
| 경도 | 약 88.7 km (37°N) | **약 8.87 m** |

즉 캐시 키 하나가 커버하는 영역 ≈ **11m × 9m 격자**.

```
       경도 →
       0.0001
  ┌────┐
  │ A  │  ← 이 격자 안의 모든 좌표는 같은 cacheKey
  │    │
  └────┘
   위도
   0.0001
```

## 시나리오별 hit / miss

### ✅ 캐시 HIT 케이스

```
1. 강남역 1번 출구 (37.4979, 127.0276)
   → 다시 강남역 1번 출구 검색
   → cacheKey 동일 → hit
   → 응답 ~50ms

2. 같은 빌딩의 1층 카페와 10층 사무실
   → 카카오 Places 가 둘 다 빌딩 중심 좌표 반환
   → cacheKey 동일 → hit

3. 광화문 (37.5717, 126.9764) 입력 후
   광화문역 1번 출구 (37.5717, 126.9763) 입력
   → 경도 4자리 반올림 둘 다 126.9764 / 126.9763 — 다를 수 있음
   → 4자리 동일하면 hit, 5번째 자리에서 갈리면 miss
```

### ❌ 캐시 MISS 케이스

```
1. 광화문 (37.5717, 126.9764)
   광화문 옆 빵집 — 30m 떨어진 위치 (37.5719, 126.9762)
   → 4자리 반올림: 37.5719 / 126.9762
   → 다른 cacheKey → miss
   → 신규 ODsay 호출 470건 발생 (~100초)

2. 강남역 1번 출구와 강남역 11번 출구 (지하 양쪽 끝)
   → 약 200m 떨어짐 → cacheKey 다름 → miss

3. 같은 도로 양쪽 빌딩 (도로 폭 20m)
   → 위도/경도 둘 다 변동 → cacheKey 다름 → miss
```

## 광화문 옆 빵집 시나리오 — 결론

```
거리 < 11m  ⇒ 캐시 hit (광화문과 정확히 같은 위치)
거리 11~20m ⇒ 50% 확률 hit (경계에 따라)
거리 > 20m  ⇒ 캐시 miss (새로운 ODsay 호출)
```

빵집이 광화문에서 30m 떨어져 있으면 → **신규 호출 발생**. 광화문 캐시는 재사용되지 않는다.

## 왜 4자리인가 — 트레이드오프

| 자리수 | 격자 크기 | hit률 | 정확도 |
|---|---|---|---|
| 3자리 | 약 110m × 89m | ⬆ 매우 높음 | ⬇ 다른 건물도 같은 캐시 (오해 발생) |
| **4자리** | **11m × 9m** | **균형** | **건물 1동 단위 정확** |
| 5자리 | 1.1m × 0.9m | ⬇ 거의 안 됨 | ⬆ 정확하지만 캐시 의미 없음 |

→ 4자리는 **"같은 빌딩 = 같은 캐시"** 라는 합리적 가정과 맞는다.

## 데이터베이스 관점

```sql
-- 캐시 키 + 행정동 코드 복합 unique
UNIQUE INDEX uniq_commute (cache_key, legal_dong_code)

-- 강남역 캐시 (1행)
cache_key='37.4979_127.0276' × legal_dong_code='1156013000' → transitMinutes=39

-- 광화문 캐시 (별도 1행)
cache_key='37.5717_126.9764' × legal_dong_code='1156013000' → transitMinutes=58
```

→ **직장 1개당 행정동 N개 행이 생긴다** (서울 전체면 ~470행).

855행 = 강남역(470) + 광화문(385) 정도가 적재된 상태라는 뜻 (광화문은 ODsay 응답 없는 일부 건너뜀).

## v1.1 — KNN 격자 확장 검색 (적용됨)

> 빵집 30m 케이스 해결을 위한 도입. ODsay 호출량 절감 효과 큼.

### 동작

```
조회 시점:  origin (lat, lng) 주변 3×3 격자 9개 키를 IN 검색
            → 약 33m × 27m 범위 내 모든 캐시 후보 회수
            → 각 행정동마다 origin 에 가장 가까운 워크포인트 캐시 선택

저장 시점:  정확 cacheKey 로만 저장 (4자리 반올림)
            → 같은 행정동에 여러 워크포인트 캐시가 누적되어 KNN 정확도 ↑
```

### 격자 9개 예시 — 강남역 (37.4979, 127.0276)

```
  ┌────────────────────────────────────────────────┐
  │ 37.4980_127.0275 │ 37.4980_127.0276 │ 37.4980_127.0277 │
  ├────────────────────────────────────────────────┤
  │ 37.4979_127.0275 │ 37.4979_127.0276 │ 37.4979_127.0277 │ ← 중심 = exactKey
  ├────────────────────────────────────────────────┤
  │ 37.4978_127.0275 │ 37.4978_127.0276 │ 37.4978_127.0277 │
  └────────────────────────────────────────────────┘
```

### 시나리오별 결과 변화 (v1 → v1.1)

| 시나리오 | v1 (정확 일치만) | v1.1 (9격자 KNN) |
|---|---|---|
| 강남역 1번 출구 → 같은 위치 다시 | hit | hit (exactMatch) |
| 광화문 → 옆 30m 빵집 | miss (신규 470건) | **nearby hit (호출 0)** ★ |
| 도로 양쪽 빌딩 (20m 폭) | miss | nearby hit ★ |
| 같은 빌딩 1층 ↔ 10층 (좌표 동일) | hit | hit |
| 200m 떨어진 다른 출구 | miss | miss (격자 ±2 이상) |

### API 응답에 추가된 필드

```json
{
  "cacheKey":    "37.4979_127.0276",
  "cacheHit":    420,    ← 정확 좌표 일치
  "cacheNearby": 30,     ← KNN 격자 흡수
  "cacheMiss":   20,     ← 신규 ODsay 호출
  "written":     20,
  ...
}
```

### 응답 데이터 구조 — `exactMatch` 플래그

서버 내부에서는 각 항목에 `exactMatch: boolean` 이 붙어 있지만,
클라이언트 응답에서는 제거 (`matrix` 안에는 transitMinutes 등 데이터만).

통계는 `cacheHit / cacheNearby / cacheMiss` 3개 카운트로 클라이언트에 전달.

### 한계

```
한계 1   격자 ±1 = 약 33m 이내만 흡수. 그 이상 거리는 여전히 miss.
한계 2   workLabel 이 첫 등록자의 라벨 유지 (광화문 캐시를 빵집이 흡수해도 label="광화문")
        → 디버그 용도라 큰 문제 없음
한계 3   같은 33m 범위에 도로/다리가 있으면 부정확 (한강대교 양쪽 등)
        → 매우 드문 케이스, v2 에서 거리 기반 KNN (B) 으로 보강 가능
```

## v2 향후 개선 방향 (아직 미적용)

```
[B] 거리 기반 KNN (정공)
    9격자 대신 모든 캐시에서 Haversine 거리 < 임계값 (50~100m) 인 항목 검색
    구현: MySQL ST_Distance / PostGIS / 또는 cache 전체 메모리 인덱싱
    장점: 격자 경계 불연속 해소

[C] 행정동 단위 정규화
    같은 행정동에 직장이 있으면 모두 같은 캐시 키 사용
    구현: 카카오 Geocoder 로 직장 좌표 → 행정동 코드 → 그걸 cacheKey 로
    단점: 행정동 안에서도 통근시간 차이 큼 (잠실동 한쪽 끝 vs 반대쪽 끝)

[D] TTL 90일 만료
    computedAt 기준 90일 지난 캐시는 자동 만료 → 도로/노선 변경 반영
    구현: 조회 시 90일 초과 항목 제외 + 백그라운드 cron 으로 삭제
```

현재 v1.1 (4자리 반올림 + 9격자 KNN) 으로 발표용 충분.
운영 단계에서 정확도 ↑ 필요하면 [B] 거리 기반 KNN 도입.

## 트러블슈팅

```
Q. 같은 직장 입력했는데 매번 ODsay 호출 발생
A. 카카오 Places 가 반환한 좌표가 미세하게 다를 수 있음
   - 검색어가 다르면 다른 매칭 → 다른 좌표 → 다른 캐시 키
   - 해결: 같은 검색어 사용 또는 퀵 칩 (인기 직장) 사용

Q. 빵집을 광화문 캐시로 처리하고 싶다
A. v1 에선 불가. [A] KNN 또는 [B] 행정동 정규화 도입 필요.

Q. 캐시 삭제는?
A. 현재 자동 만료 없음. 수동 SQL:
   DELETE FROM t_commute_matrix WHERE cache_key = '37.4979_127.0276';
```
